## PR-staleness-floor — Correctness invariant + sensor audit

**Item 6** — staleness must be ≥ reseed cadence for reseed-driven sensors. The user's literal rule "staleness ≥ cadence" cannot apply universally without breaking the matrix's deliberate `staleness ≪ reseed` on fast signal-driven paths. The plan splits the invariant by regime.

### Invariant (lifted from cadence matrix §"Relationship between staleness…")

For every sensor we must satisfy `staleness > max(organic_signal_gap, reseed_cadence)` in the regime that applies. Concretely:

| Classification | Staleness rule | Why |
|---|---|---|
| Fast (organic ≥1 Hz) | `staleness ≥ 2 × max_organic_gap` (~5 s typical) | Signal drives freshness; fail-fast on signal loss is desired. |
| Slow-signalled | `staleness ≥ 2 × organic_gap` AND `staleness ≥ 2 × reseed_cadence` | Both regimes apply; take the larger. |
| Reseed-driven / readback | `staleness ≥ 2 × reseed_cadence` | Reseed IS the freshness source. |

### Audit table (every sensor under current code)

`SensorId::freshness_threshold` vs `SEED_INTERVAL_DEFAULT=60s` / `SEED_INTERVAL_SETTINGS=300s`:

| Sensor | Class | Cadence | Staleness | OK? | Action |
|---|---|---|---|---|---|
| BatterySoc | Slow-signalled | 60 s | **15 s** | **VIOLATION** | Raise to 120 s (2 × 60 s reseed) |
| BatterySoh | Reseed-driven | 60 s | 900 s | OK | none |
| BatteryInstalledCapacity | Reseed-driven | 60 s | 3600 s | OK | none |
| BatteryDcPower | Fast | 60 s | 5 s | OK (fast regime) | none |
| MpptPower0/1 | Slow-signalled (idle at night) | 60 s | 30 s | borderline | acceptable; matrix-justified |
| SoltaroPower | Fast | 60 s | 5 s | OK | none |
| PowerConsumption / ConsumptionCurrent | Fast | 60 s | 5 s | OK | none |
| GridPower / GridCurrent | Fast | 60 s | 5 s | OK | none |
| GridVoltage | Fast (slow-moving) | 60 s | 10 s | OK | none |
| OffgridPower / OffgridCurrent / VebusInputCurrent | Fast | 60 s | 5 s | OK | none |
| EvchargerAcPower / EvchargerAcCurrent | Fast | 60 s | 5 s | OK | none |
| EssState | Reseed-driven | 300 s | 900 s | OK | none |
| OutdoorTemperature | Reseed-driven (Open-Meteo 30 min) | 1800 s | 2400 s | OK | none |
| **Readback** InputCurrentLimit | Readback | 60 s | 600 s | OK | none |
| **Readback** GridSetpoint / Schedule0/1 fields | Readback | 300 s | 900 s | OK | none |

**Net offenders: 1** (BatterySoc — exactly the case the user named).

### Implementation

1. **Per-variant invariant test**: add a `#[test]` that iterates every `SensorId` variant and asserts the `staleness ≥ 2 × cadence` rule in the appropriate regime. Fast-class sensors get a separate compile-time list (otherwise the rule misfires on legitimately-short fast windows).
2. **Startup assertion**: a one-shot call from `runtime.rs::Runtime::new` walks every `SensorId` and panics on violation with a clear message naming the offender.
3. **Fix BatterySoc**: change `Self::BatterySoc => Duration::from_secs(15)` to `Duration::from_secs(120)` in `crates/core/src/types.rs`. Rationale: SoC is slow-signalled at ~1 Hz when changing, minutes idle. With 60 s reseed, 120 s = 2 × reseed.
4. **Update the cadence matrix doc** to note the BatterySoc revision.

### Acceptance

- All sensor variants pass the new test.
- `BatterySoc` no longer flips Stale during quiet 60-90 s SoC plateaus.
- Startup-assertion test forces failure when one of the constants is hand-edited to violate the rule.

### Deferrable

Whether to relax controllers' "Stale = not usable" stance is a **separate design call** — not in this PR. Flagged as Open Question Q1 below.

## PR-ha-discovery-expand — All sensors + bookkeeping to HA

**Item 4.** Extend `crates/shell/src/mqtt/discovery.rs` and add per-sensor publishing.

### Surface

- 19 D-Bus sensors + 1 `outdoor_temperature` + 1 new `session_kwh` = 21 HA `sensor` entities.
- Bookkeeping fields **worth exposing** (numeric/bool, controller-relevant): `zappi_active` (bool), `charge_to_full_required` (bool), `soc_end_of_day_target` (%), `effective_export_soc_threshold` (%), `battery_selected_soc_target` (%), `charge_battery_extended_today` (bool), `prev_ess_state` (int).
- **Skip-by-default** (date-string only): `next_full_charge_iso`, `above_soc_date_iso`, `charge_battery_extended_today_date_iso`, `last_weather_soc_run_date`. → Open Q2 below.

### Discovery payload table

| Sensor | unit_of_measurement | device_class | state_class |
|---|---|---|---|
| battery_soc, battery_soh, *_soc_target, *_soc_threshold | `%` | `battery` (soc/soh), none for thresholds | `measurement` |
| session_kwh | `kWh` | `energy` | `total_increasing` |
| *_power, mppt_power_*, soltaro_power, grid_power, offgrid_power, evcharger_ac_power | `W` | `power` | `measurement` |
| *_current, vebus_input_current, evcharger_ac_current | `A` | `current` | `measurement` |
| grid_voltage | `V` | `voltage` | `measurement` |
| outdoor_temperature | `°C` | `temperature` | `measurement` |
| ess_state | none | none | `measurement` |
| Bookkeeping booleans | — | — | published as `binary_sensor` (component=`binary_sensor`, payload_on=`"true"`) |

### Topics

- State: `<topic_root>/sensor/<name>/state` (single string body, NOT JSON — matches existing knob pattern).
- Discovery config: `<HA_ROOT>/sensor/<NODE_ID>/sensor_<name>/config` (or `binary_sensor/...`).

### Per-tick publishing

Extend `PublishPayload` with `Sensor { id: SensorId, value: Option<f64>, freshness: Freshness }` and `BookkeepingSensor`; emit from a new `SensorBroadcastCore` (no `depends_on`; runs after derivations). Existing publish dedup prevents flooding. Stale → `"unavailable"` (HA convention).

### Volume estimate

- Discovery configs (retained): ~26 new topics × ~400 B JSON = **~10 KB retained** on top of existing ~16 KB. Total ~26 KB. FlashMQ on Venus handles megabytes — comfortable.
- State publishes: dedup'd to "publish only on change" → realistic 5-15 publishes/s during active periods.

### Files

- `crates/core/src/types.rs::PublishPayload` (+ `Sensor`, `BookkeepingSensor`)
- `crates/core/src/core_dag/cores.rs` (+ `SensorBroadcastCore`)
- `crates/shell/src/mqtt/serialize.rs::encode_publish_payload` (+ arms; staleness = `"unavailable"`)

## Cross-cutting architectural notes (lock in tasks.md)

- **Honesty invariant**: PR-tass-dag-view's outcome tracking does not suppress per-controller Decision writes.
- **Three-layer safety chain** preserved: HA discovery only adds read-only `sensor`/`binary_sensor`s; no new writable entities.
- **Wire format**: 0.1.0 → 0.2.0, additive only; no removals.
- **Description registry**: TS-only. Different audiences from HA discovery.
- **MQTT volume**: ~26 KB total retained after expansion; FlashMQ default tolerances comfortably accommodate.
- **No opportunistic refactors**.

## Risk list

1. **HA retained-message inflation**: ~26 KB total retained — below FlashMQ defaults.
2. **HA "unavailable" semantics**: verify no current HA automation relies on absence of unavailability.
3. **Wire-format bump**: TS dashboard regenerates from baboon; no external clients besides HA.
4. **Description-registry drift**: a frontend-only registry will go stale when entities are added — flag as follow-up to add CI completeness check.
5. **TASS DAG outcome semantics**: today every core runs every tick, so "outcome" is always success. Risk: vestigial field. Mitigation: ship as planned; revisit when actual failure paths exist.
6. **BatterySoc 15 → 120 s**: widens the silence window. User-requested + matrix-anchored.
