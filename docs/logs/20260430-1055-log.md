## Round-by-round defect highlights

### PR-ZDO-1 round 1 — major control-flow defect caught

**D01 (major)**: `run_setpoint` runs on every event in production
(sensor readings, typed readings, schedule readbacks, etc.), not just
on Tick. Naive capture-every-call would have collapsed the 120-sample
30-min buffer to seconds. The chart's "30 min" label would have been a
lie within minutes of deployment.

**Fix**: time-gate the `samples.push_back` half of `push` on
`SAMPLE_INTERVAL_MS = 15_000`. `latest` updates unconditionally on
every call so HA broadcasts (PR-ZDO-2) and wire snapshots (PR-ZDO-3)
stay lockstep with the controller's per-event output. Round-2 review
caught two follow-ups (D08 weak `latest`-on-gated-call assertion, D09
missing doc on backwards-clock-jump behaviour); both resolved.

This is the second time in two milestones that round-1 review caught a
locked-design defect — **the pattern of "plan-faithful execution that
silently violates production semantics" is the highest-value review
target**. Future plans should explicitly note the production-runtime
characteristics that constrain the spec (e.g., "process() runs on
every event, not just Tick").

### PR-ZDO-2 round 1 — two HA-recorder integrity defects

**D01 (major)**: HA discovery configs included
`availability_topic: "<base>/availability"` — a topic the controller
never publishes to. HA would have rendered all three new entities as
permanently "Unavailable", regardless of state-topic content.

**D02 (major)**: Disabled-branch placeholder `compensated_drain_w =
0.0` (locked semantic from PR-ZDO-1 — the controller didn't run; the
zero is a stand-in) was being broadcast as a real 0 W reading. HA
Recorder would have ingested fake zeros during every
`apply_setpoint_safety` fallback. The user's framing — "a derived
sensor out of sync with the controller is worse than no observability"
— would have been violated.

**Fix**: removed `availability_topic` (inline `unavailable` token via
`encode_sensor_body` is the project convention); added a guard arm in
the broadcast match: `Some(s) if s.branch == Disabled` → `(0.0,
Stale)` so the wire body encodes as `"unavailable"`, just like the
no-snapshot case. Both surfaces now respect the same skip-Disabled
contract.

The two majors demonstrate that **lockstep is operationally about more
than just the value-axis** — the HA-side surface (recorder, dashboards)
is itself a downstream renderer that must respect the same contract as
the in-dashboard chart.

### PR-ZDO-4 round 1 — test rigor gap (typical for this codebase)

The reviewer flagged that the three "PR-ZDO-4.T*" tests verify only
the constant lookup tables, not the renderer outputs themselves
(D01). This is structural to the project's "tsc + esbuild only, no
test runner" stance and is consistent with how `fmtMpptOperationMode`
was previously tested. Mitigated by extracting a pure
`summaryFor(latest)` decision helper that's now properly unit-tested
(22 assertions across 4 scenarios including the Disabled→`—` honest
contract).

## Cross-cutting lessons

- **Plan-faithful execution can violate production semantics.** Two
  rounds, two PRs (PR-ZDO-1 + PR-ZDO-2), both caught major defects on
  round 1 because the executor implemented the plan exactly while the
  plan didn't note an operational constraint. Future plans should call
  out: per-event vs per-tick callback frequencies, downstream surfaces
  (HA recorder) that count as renderers, and which knobs / fields are
  on the broadcast-vs-wire-vs-control path.
- **Lockstep is multi-axis.** Value lockstep (broadcast ⇔ controller
  output): straightforward; achieved by reading `world.zappi_drain_state.latest`
  directly. Time lockstep (samples-per-real-time-window ⇔ chart label):
  needed PR-ZDO-1's gate fix. Surface lockstep (HA recorder ⇔
  in-dashboard chart): needed PR-ZDO-2's Disabled-branch fix. All three
  axes need explicit verification.
- **Pure decision helpers enable testability without test runners.**
  Extracting `summaryFor(latest)` from `renderZappiDrainSummary` made
  the rendering decision logic unit-testable in `web/src/render.test.ts`.
  Pattern is now reusable: any future DOM-mutating renderer should
  separate decision from poking.
- **Topic namespaces matter.** Reserved `controller/<name>/state` as
  the prefix for controller-derived broadcasts. Future siblings
  (setpoint decision tag, schedule activation flags) ride this prefix
  cleanly. Locked in `tasks.md` cross-cutting note.

## Constraints captured for future work

- The `// LOCKSTEP:` comment cross-references between
  `classify_zappi_drain_branch` and `evaluate_setpoint`'s if/else
  ladder MUST be maintained. PR-ZDO-1.T3 is the regression guard.
- No controller branch reads from `world.zappi_drain_state`.
  PR-ZDO-1.T6 locks this.
- `SAMPLE_INTERVAL_MS = 15_000` matches default tick cadence; if tick
  rate ever changes, this constant must move with it.
- `branch == Disabled` is the canonical "controller couldn't run"
  signal. Any future broadcast / chart surface MUST treat Disabled as
  Stale, not as a real reading.
- The `controller/<name>/state` MQTT topic root is reserved for
  controller-derived observables. HA discovery for these MUST NOT
  include `availability_topic` unless the controller actually
  publishes to it.
- The `WorldSnapshot.zappi_drain_state` field MUST stay at the end of
  the WorldSnapshot baboon block (additive-within-v0.3.0 invariant).
- The `captured_at_ms` (core) → `captured_at_epoch_ms` (wire) name
  swap is intentional convention.
- The Disabled→`—` invariant in `summaryFor` MUST be preserved. A
  "fix" that displays `0 W` for Disabled would re-introduce the
  dishonest-zero defect that PR-ZDO-2-D02 fixed.
- Future renderers consuming `ZappiDrainState` MUST sort `samples` by
  `captured_at_epoch_ms` at render time (GX clock can jump backwards).
