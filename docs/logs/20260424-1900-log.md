## Notable discoveries

- **`RealClock::naive()` is uncached.** Two calls within a tick
  can return different wall-clock values. PR-DAG-A round-1 made
  `compute_derived_view` run twice per tick (once per consumer
  core), which would have re-introduced A-05 across the 5-min
  WAIT_TIMEOUT_MIN boundary for the Zappi WaitingForEv path.
  Caught by the round-1 reviewer and fixed by adding
  `&DerivedView` to `Core::run`, then deleted entirely in -B.
- **Observer-mode target mutation shape.** PR-05 chose "don't
  mutate target in observer mode" as belt-and-suspenders for
  A-06 (stuck-Pending). That made observer mode uninformative
  on the dashboard. PR-SCHED0 reverts half: target is set;
  effect emission stays gated. A-06 still prevented via
  PR-05's KillSwitch(false→true) edge-reset on all six targets.
- **`propose_target` was not side-effect-free.** It called
  `self.actual.deprecate(now)` inside. Moved to `mark_commanded`
  (the "we just emitted the write" signal) — cleaner TASS contract,
  observer mode no longer taints Actual's freshness machine.
- **PR-04's bookkeeping latch was hiding sensor loss.** When both
  the Zappi typed state AND `evcharger_ac_power` went Stale,
  `run_current_limit` would early-return on the freshness gate
  and leave `bk.zappi_active` at its last true value; schedules
  would then read that latched value. PR-DAG-B's
  `ZappiActiveCore` runs unconditionally and drops to false on
  unusable evidence — more honest, safer.
- **Double `Publish(ActuatedPhase)` per live-path tick.** Post
  PR-SCHED0 the live path emits Publish(Pending) pre-gate AND
  Publish(Commanded) post-`mark_commanded`. At 1 Hz that's ~172k
  publishes/entity/day on an external broker. Tracked as
  PR-SCHED0-R2-D04; deferred to M-AUDIT-2 MQTT hygiene sub-PR.
