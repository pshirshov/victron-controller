# Session log — 2026-05-01 21:24 — M-EDDI-SENSORS

## Original user request

The conversation opened with a field observation: dashboard reported
`eddi.mode.target = Stopped, EddiController, Confirmed, actual =
Stopped` while HASS's separate myenergi integration showed the Eddi
in Normal and the device was physically diverting. Investigation
identified two distinct issues:

1. **Parser bug** in `parse_eddi` (`sta=1` → Normal, else Stopped),
   conflating myenergi operational status with mode.
2. **Visibility gap** — typed sensors (eddi mode, zappi state) were
   not in the dashboard sensors table, only surfaced as the actuated
   row's "actual" column, which was contaminated by the parser bug.

After scoping discussion, the user asked for the four-part work to
be implemented via `/review-loop`:

- A. Tighten `parse_eddi` heuristic per documented `sta` codes.
- B. Add eddi + zappi to the dashboard sensors list (forecasts and
  D-Bus values explicitly out of scope).
- C. Capture last raw `cgi-jstatus-{Z,E}` JSON body in-memory; latch
  on success only, no persistence beyond restart.
- D. Entity inspector popup shows the raw response with a copy
  button; degrades silently when no body is captured yet.

## Notes for future work

- `parse_eddi` mapping remains heuristic-from-docs. Once the user
  deploys this PR, clicking the eddi.mode sensor in the popup will
  surface the actual `sta` value seen on the user's firmware. If a
  firmware variant emits a Stopped indicator outside `{0, 6}`, refine
  the parser then.
- The boot-time "since" timestamp issue (`Actual::unknown(now)` stamps
  the boot Instant) applies to every f64 sensor row in the dashboard.
  Fixed only for the new typed-sensor rows in this PR; sweeping the
  f64 rows is a separate UX-tightening PR.
- The "raw body intentionally outlives freshness decay" invariant is
  now load-bearing in `apply_typed_reading`. Any future
  `TypedReading::{Eddi,Zappi}` emitter that doesn't carry a fresh
  body must use `raw_json: None` to leave the latch alone, not to
  clear it.
- Inspector copy buttons for multi-line text should use the
  `data-copy-from-sibling="true"` marker pattern, not
  `data-copy="..."`. The latter is broken for any string containing
  `"`, since `esc()` only escapes `&`, `<`, `>`.
